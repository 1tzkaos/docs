---
title: "Wallet-intel tags"
description: "What each wallet tag means, how it's computed, and where it appears across the API."
---

Wallet-intel attaches cohort tags to wallets so you can filter, score, and visualize "what kind of holder is this". The same tag list shows up on [`/wallets/{addr}/tags`](/api-reference/wallets/getwallettags), embedded under `tags` on holders and snipers in [`/tokens/{mint}/intel`](/api-reference/tokens/gettokenintel), and aggregated into counts at [`/tokens/{mint}/wallet-tag-counts`](/api-reference/tokens/getwallettagcounts).

## The tags

| Tag | Meaning |
| --- | --- |
| `dev` | The wallet that created the mint. Always a single wallet per mint; checked against the token's creator field. |
| `fresh` | Wallet created recently (typically within the last few days) and has very few historical trades. Useful for spotting wash-trading or coordinated buys. |
| `insider` | Wallet currently holds a non-zero balance but has never traded the mint on a DEX. The classic "they got it some other way" signal. |
| `kol` | Wallet is on a curated known-influencer list. Source list is maintained internally; no public submission API yet. |
| `smart_money` | Wallet has a positive realized-PnL track record across enough trades to clear a confidence threshold. |
| `sniper` | Wallet appears in the first 10 distinct buyers within the first 30 slots after the mint was created. Mint-scoped (a wallet can be a sniper for token A but not token B). |
| `whale` | Wallet's lifetime SOL-denominated volume crosses a whale threshold. Global. |

## Scope

Some tags are global (apply to the wallet across every mint): `dev`, `fresh`, `kol`, `smart_money`, `whale`. Others are mint-scoped (apply only to a (wallet, mint) pair): `insider`, `sniper`.

When you fetch `/wallets/{addr}/tags`, mint-scoped tags carry the `mint` field that scopes them; global tags carry the wallet's principal mint or an empty mint placeholder — key off the tag name, not the mint, when deciding whether a tag is global.

## Recomputation

The snipers and insiders sets recompute lazily — the first read after a long idle period may return empty arrays while a background job fills them in. Read the endpoint twice spaced a few seconds apart if you see suspicious emptiness for a mint you know has activity.
